This plugin renders documents with poi-tl (poi template language). Current version: 1.12.2.
See template syntax details at: poi-tl documentation
  1. Available base variables:
    1. {{title}} document title
    2. {{author}} author
    3. {{date}} current date
    4. {{version}} version
    5. {{toc}} table of contents placeholder
  2. Available collection variables:
    1. {{dataSources}} selected data sources (databases and objects):
      1. das raw datasource object
      2. name datasource name
      3. comment datasource comment
      4. dialect datasource dialect
      5. version datasource version
      6. namespaces namespace list (usually schema):
        1. das raw namespace object
        2. name namespace name
        3. comment namespace comment
        4. tables object list (tables/views/materialized views):
          1. das raw object
          2. name object name
          3. comment object comment
          4. objectKind type (TABLE/VIEW/MATERIALIZED_VIEW)
          5. schema schema name
          6. primaryKey primary key
          7. columns column list (name/comment/type/defaultValue/isPrimaryKey, etc.)
    2. For views/materialized views with missing column comments, comments are backfilled from same-name table columns when possible.
    3. {{ds4table}} is equivalent to {{dataSources}} (for LoopRowTableRenderPolicy).
    4. {{ns4table}} is equivalent to namespaces (for LoopRowTableRenderPolicy).
    5. Object-kind section variables:
      1. Tables: {{dataSourcesTables}} / {{namespacesTables}} / {{ds4tableTables}} / {{ns4tableTables}}
      2. Views: {{dataSourcesViews}} / {{namespacesViews}} / {{ds4tableViews}} / {{ns4tableViews}}
      3. Materialized Views: {{dataSourcesMaterializedViews}} / {{namespacesMaterializedViews}} / {{ds4tableMaterializedViews}} / {{ns4tableMaterializedViews}}
    6. For large datasets, the plugin renders docx parts in parallel, merges them, and cleans temporary files automatically.
  3. Sort feature (optional):
    1. Click Sort Config in the bottom-left area of the main dialog to expand the left panel.
    2. Sort source options:
      1. Disable Sort: keep original order (default).
      2. Multi-line Input: one table name per line; empty lines and # comments are supported.
      3. TXT File: load a UTF-8 text file with one table name per line.
      4. Drag in List: the list shows all table names, initialized in first-seen metadata order, with the same name de-duplication as multi-line/TXT. Multi-select a block and drag and drop it to a new insert position. The number. prefix on each line is for display only; the actual order key is still the table name string.
    3. Rule: matched tables are ordered by your list; unmatched tables are appended after them.
    4. Sorting is case-insensitive; duplicate table names are deduplicated (first occurrence wins).
    5. If the same table name exists in different schemas, ordering uses the table name as the key (same as multi-line, TXT, and the list), which may be ambiguous; adjust your expectations accordingly.
  4. Plan variables quick reference (poi-tl):
    1. Core plan variables:
      1. {{plan}}: full plan tree (root list in plan.roots).
      2. {{vars}}: compatibility map; parent variable value is List<DocNode>, leaf variable value is DocNode.
      3. {{varNodes}}: unified node map; both parent and leaf variables resolve to DocNode.
      4. {{varItems}}: unified list map; both parent and leaf variables resolve to List<DocNode> (leaf variables are single-item lists).
    2. Prefer {{varItems.xxx}} for loop rendering (same semantics for parent and leaf variables).
    3. Prefer {{varNodes.xxx}} for node metadata (display name, type, variable name, etc.).
    4. Common DocNode fields:
      1. displayName
      2. type (GROUP / DATABASE / NAMESPACE / TABLE / VIEW / MAT_VIEW)
      3. variableName
      4. order
      5. enabled
      6. ref.dataSource / ref.namespace / ref.objectName / ref.objectKind
    5. Semantics:
      1. If a node is disabled (enabled=false), it is excluded from vars / varNodes / varItems.
      2. For a parent variable, varItems represents all enabled descendant leaf nodes.
      3. For a leaf variable, varItems is a single-item list, which keeps templates consistent.
    6. Examples:
      1. Group title: {{varNodes.core_group.displayName}}
      2. Group loop: {{#varItems.core_group}} ... {{displayName}} / {{type}} ... {{/varItems.core_group}}
      3. Single-table unified loop: {{#varItems.table_user}} ... {{ref.namespace}} ... {{/varItems.table_user}}
      4. Full tree traversal: {{#plan.roots}} ... {{displayName}} ... {{#children}} ... {{/children}} {{/plan.roots}}
